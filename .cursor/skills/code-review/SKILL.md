# 代码检视 (Code Review)

## 快速指南

当用户提供 diff 文件并要求进行代码检视时，请严格遵循以下工作流程：

### 1. 总结变更内容
首先，分析 diff 文件，向用户简要总结本次代码修改了哪些主要内容。

### 2. 逐文件检视
逐个文件检查代码修改，必须重点关注以下检查点：

- **模块级接口变更**：接口的变更是否合理？是否破坏了向后兼容性？
- **功能问题**：代码逻辑是否存在缺陷或潜在 bug？
- **代码简洁与优化**：代码是否冗余？是否有更好的实现方式或优化空间？
- **推理性能影响**：该修改是否可能导致程序（特别是模型推理过程）性能下降或增加耗时？
- **架构影响**：修改是否破坏了整体架构设计或分层原则？
- **特定语言风险写法**：
  - **C/C++**：是否存在数组/内存的**越界访问**风险？是否存在**空指针解引用**的可能？是否存在未释放内存或文件句柄导致的**资源泄漏**？
  - **Python**：是否存在可变默认参数导致的陷阱？全局变量的滥用？循环中不合理的性能开销？是否存在因为深浅拷贝导致的数据篡改风险？
  - **其他语言**：根据该语言的最佳实践指出类似的常见陷阱与风险写法。
- **文件权限问题**：涉及文件读取或写入时，是否存在权限请求不当或权限校验缺失的问题？

### 3. 反馈格式

对于发现的每一个问题，**必须严格使用**以下格式进行输出：

```text
[问题分类]: <描述问题分类点，例如：特定语言风险写法/空指针、功能问题、架构影响等>
[具体问题]: <具体描述问题所在，有什么风险>
[修改建议]: <给别人提供具体的修改建议或代码示例>
```

## 检视示例 (Example Output)

**Diff 变更总结**:
本次修改主要在 `src/inference.cpp` 中增加了一个模型加载和文件读取逻辑，并修改了 `scripts/process_data.py` 中的数据预处理函数接口。

**代码检视反馈**:

**`src/inference.cpp`**
[问题分类]: 特定语言风险写法 (C++)
[具体问题]: 第45行 `buffer` 数组在使用前未进行大小和边界检查，如果读取的模型文件过大，会导致内存越界访问风险，甚至引发进程崩溃。
[修改建议]: 在读取数据到 `buffer` 之前，添加对文件大小的校验，或者推荐使用 `std::vector<char>` 来动态管理内存。

[问题分类]: 文件权限问题
[具体问题]: 第60行 `open(filepath, O_RDWR)` 请求了读写权限，但后续逻辑仅包含读取操作。如果目标模型文件在文件系统中为只读，此操作会导致文件打开失败。
[修改建议]: 建议将权限标志修改为只读 `O_RDONLY`。

**`scripts/process_data.py`**
[问题分类]: 特定语言风险写法 (Python)
[具体问题]: `process_batch` 函数使用了一个可变的列表作为默认参数 (`def process_batch(data, results=[]):`)，这将导致在多次调用该函数时共享同一个列表对象，产生意料之外的数据累加问题。
[修改建议]: 将默认参数改为 `None`，并在函数内部进行初始化，例如：`if results is None: results = []`。

**`include/model.h`**
[问题分类]: 模块级接口变更
[具体问题]: `initModel` 函数增加了一个新参数 `bool use_cache`，但没有提供默认值。这将导致所有旧的调用方代码编译报错，破坏了向后兼容性。
[修改建议]: 为 `use_cache` 参数提供一个合理的默认值（例如 `bool use_cache = false`），或者保留旧接口并通过函数重载实现新功能。
